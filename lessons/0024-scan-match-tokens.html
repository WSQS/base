Lesson 0024

先让 scanner 认识 match 相关 token

这一课不做完整的模式匹配，只做词法层的最小准备：让 scanner 先认识未来最小 match 表达式所需的 token。

这一步为什么重要： 你已经明确一节课默认只修改一个实现阶段。既然未来的 match 会同时牵动 scanner、parser、AST 和 evaluator，那第一步就应该先只停留在 scanner 层。

本课范围

  • 为未来最小 match 引入必要 token
  • 至少考虑这些符号或关键字：match、_、=>
  • 如果你希望未来最小 match 采用花括号和 arm 分隔符，也应一起进入 scanner，例如 {、}、,
  • 补上 scanner 层面的测试

这一课不做：

  • 不做 Pattern AST
  • 不做 match parser
  • 不做 evaluator 匹配执行

这节课真正新增的知识点

这节课的重点不是模式匹配语义，而是：

未来一个新语法结构进入语言时，最前面的那一层应该先认识它的符号。

也就是说，现在先解决：

  • match 不再被扫成普通标识符
  • _ 是否成为专门 token
  • => 是否作为双字符 token 被识别
  • 如果未来语法写成块状分支，{、}、, 是否也进入 token 序列

推荐的最小实现方向

  1. 先确定未来最小 match 语法到底要哪些词法符号
  2. 在 Token 中加入相应变体
  3. 修改 scan() 支持这些 token
  4. 新增 scanner 测试，验证 token 序列正确

这一课的重点不是“match 已经能运行”，而是“match 已经能被语言看见”。

目标输入 / 输出

未来完整示意：
    match true {
        true => 1,
        false => -1,
        _ => 0
    }

    本课 scanner 目标：
    先让这类写法涉及到的关键符号能够被正确切成 token。

注意：这里只验证 token 序列，不要求 parser 或 evaluator 已经理解这整个 match 表达式。

  • 新增 scanner 测试并通过

提醒： 这节课的胜利条件不是模式匹配跑通，而是“未来完整 match 写法所需的关键语法符号已经进入词法层”。
参考来源：
