Lesson 0018

先引入 Value 类型

这一课只教一件事：让解释器的运行时结果不再直接等于整数，而是统一通过一个 Value 类型来表示。

和你的 mission / language 的关系： 你已经让这门语言拥有了表达式、变量环境和更稳定的 parser。现在如果想继续往布尔值、比较运算、条件语句甚至更真实的代码解释能力发展，当前“表达式求值结果就是裸 i64”这个前提会开始卡住你。先引入 Value 类型，是在不急着加新语法的前提下，为后续值语义扩展打地基。

今天只做这一件事

这节课不要同时做布尔值、比较运算和条件语句。范围严格控制在：

  • 定义一个运行时值类型，例如 enum Value
  • 让 eval_expr() 返回 Value，而不是直接返回 i64
  • 让变量环境保存 Value
  • 让当前已有的整数算术在 Value 上继续正确工作

第一版即使 Value 里暂时只有：

Value::Integer(i64)

也完全有意义，因为它已经把运行时接口从“裸整数”提升成了“统一值模型”。

这节课真正发生的变化

在这一课之前，解释器大致默认：

expression result = i64

这一课之后，你要把它改成：

expression result = Value

这看起来像小改动，但它其实是在改变解释器的核心接口。后面加布尔值、比较表达式、条件控制流时，就不需要再从头推翻 evaluator 的返回模型。

推荐的最小实现方向

  1. 先定义 enum Value
  2. 把 eval_expr() 的返回值从 i64 改成 Value
  3. 把环境从 HashMap<String, i64> 改成 HashMap<String, Value>
  4. 把整数算术分支改成在 Value::Integer 上工作
  5. 让 print 能输出当前已有的整数值

这一课的关键不是把类型系统做复杂，而是先让“运行时值”有一个统一外壳。

本课目标输入 / 输出

输入程序:
print 1 + 2;

目标：
仍然输出 3
但 evaluator 内部已经返回 Value 而不是裸 i64
输入程序:
let x = 5;
print x;

目标：
环境里保存的是 Value
最终仍然输出 5

完成标准

  • 解释器已经引入 Value 类型
  • eval_expr() 返回 Value
  • 环境保存 Value
  • 当前整数程序仍然可以正确运行

这节课故意不做什么

  • 先不引入布尔字面量
  • 先不引入比较运算
  • 先不引入 if / while
  • 先不扩展成完整类型系统

这节课只做一件事：把运行时结果统一封装成 Value。

卡住时优先检查这 4 件事

  • 你是不是只定义了 Value，但没有把 eval_expr() 的返回值一起改掉？
  • 你是不是环境里还在存 i64，导致 evaluator 和环境类型不一致？
  • 你是不是在整数算术分支里还直接返回裸整数，而不是 Value::Integer？
  • 你是不是一口气开始加布尔值和比较运算，把课题做大了？
本课后的观察点： 完成这一课后，你的语言表面能力几乎没变，但内部运行时接口已经更适合继续长。后面再引入布尔值或比较表达式时，Value 模型会成为非常自然的承载点。
参考来源：
  • Programming Languages: Application and Interpretation：值模型扩展前，先把解释器的返回接口整理干净，是非常自然的语言实现步骤。
  • Essentials of Programming Languages：每加入一种语言能力，往往都意味着值语义和求值模型需要一起升级。
