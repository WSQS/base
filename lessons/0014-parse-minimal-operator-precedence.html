Lesson 0014

先让乘除比加减更紧

这一课只教一件事：让 parser 不再把所有运算符都按同一层处理，而是先做到最小优先级规则：* / 比 + - 更紧。

和你的 mission 的关系： 你已经能解析最小二元表达式和括号分组，但如果没有优先级，表达式系统还不够像真正的语言。Phase 0 想成为稳定可运行的语言原型，就必须让最常见的数学表达式不再被错误分组。

今天只做这一件事

本课不要直接上完整 Pratt parser 或完整 precedence climbing。范围严格控制在：

  • 让 * / / 的绑定强于 + / -
  • 至少正确处理 1 + 2 * 3
  • 保持已有括号支持仍然成立
  • 仍然接进现有的 let / print

也就是说，这节课的最小目标是：

print 1 + 2 * 3;

应该被解析成：

1 + (2 * 3)

而不是：

(1 + 2) * 3

这节课真正要学会的不是“多写几个 if”

你要建立的是一个新的 parser 直觉：

  • 表达式不是一层平铺
  • 不同运算符有不同“紧密程度”
  • parser 需要按层次解析表达式，而不是看到一个操作符就立刻拼一个 Binary

这正是 precedence climbing 和 Pratt parser 要解决的核心问题。只是这一课我们先做最小版，不要求一步到位。

推荐的最小实现方向

当前最自然的最小重组是把表达式解析拆成两层：

  1. 一层负责解析“更紧的部分”，例如原子表达式 / 括号 / 乘除
  2. 另一层在上面继续解析加减
  3. 这样当看到 1 + 2 * 3 时，右边会先被解析成 2 * 3

你不必一开始就用术语命名得很标准，但结构上最好开始出现“分层解析表达式”的意识。

本课目标输入 / 输出

输入程序:
print 1 + 2 * 3;

目标 AST 形状:
Binary(
  Number(1),
  Plus,
  Binary(Number(2), Star, Number(3))
)

最终输出:
7
输入程序:
print (1 + 2) * 3;

目标：
括号分组仍然优先于默认优先级
最终输出 9

完成标准

  • 你已经让 * / 比 + - 更紧
  • 你已经验证 1 + 2 * 3 不会被错误解析成 (1 + 2) * 3
  • 你已经验证括号仍然有效，例如 (1 + 2) * 3

这节课故意不做什么

  • 先不追求完整 Pratt parser
  • 先不追求任意多优先级层级
  • 先不做一整套 parser 错误恢复
  • 先不扩展新语句或新数据类型

这一课的目标只是让表达式第一次真正开始“分层”。

卡住时优先检查这 4 件事

  • 你是不是依然在看到第一个操作符时就立刻组装 Binary？
  • 你是不是还没有把表达式解析拆出至少两层？
  • 你是不是修好了 `1 + 2 * 3`，却把括号支持弄坏了？
  • 你是不是一开始就想实现完整通用表达式框架，而不是先做最小优先级？
本课后的观察点： 完成这一课后，你的 parser 就不再只是“支持括号和最小 binary”，而是第一次具备真正的运算符层次感。这会让后面继续扩展表达式系统时稳定很多。
