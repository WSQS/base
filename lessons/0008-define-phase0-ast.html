Lesson 0008

先定义 Phase 0 的最小 AST

这一课只教一件事：在真正写 parser 之前，先把 Phase 0 语言要产出的最小 AST 结构定义清楚。

和你的 mission 的关系： 你已经完成了 Phase 0 的 lexer。现在 parser 不能直接“边写边猜”，而要先明确：这门语言的程序、语句、表达式最终长成什么样。AST 就是 lexer 和 evaluator 之间的桥梁。

今天只做这一件事

这节课不要求你马上把 parser 全写完，也不要求你执行程序。你只需要先把这门语言的最小结构表示清楚。

当前 Phase 0 只需要两类语句：

  • let name = expr;
  • print expr;

表达式当前只需要覆盖：

  • 数字字面量
  • 变量名
  • 二元运算 + - * /
  • 括号分组

你要亲手定义的最小结构

推荐你把 AST 至少拆成三层：

Program
  └── Vec<Stmt>

Stmt
  ├── Let { name, expr }
  └── Print { expr }

Expr
  ├── Number(...)
  ├── Ident(...)
  ├── Binary { left, op, right }
  └── Grouping(...)

这里的重点不是语法细节，而是先回答：parser 解析完以后，要把源码变成什么样的数据结构。

你要亲手完成的步骤

  1. 先定义一个程序层结构，表示“一整个程序由多条语句组成”
  2. 定义 Stmt，只支持 Let 和 Print
  3. 定义 Expr，至少支持数字、标识符、二元运算、分组
  4. 暂时不实现真正 parser，只验证：你能否用这些类型表达两个示例程序
  5. 给 AST 类型补一个最小测试或构造样例，确认结构够用

完成标准

如果你已经做到下面两件事，这节课就完成了：

  • 定义出能表示 Phase 0 两类语句和基础表达式的 AST 类型
  • 能用这套 AST 结构表达至少一个小程序，例如 let x = 1 + 2; print x;

卡住时优先检查这 3 件事

  • 是不是把语句和表达式混在一起了？
  • 是不是还在用 token 序列思考，而没有切换到“结构树”的视角？
  • 是不是试图一步到位把所有未来特性都塞进 AST？
本课后的观察点： 完成这一课后，下一步 parser 的目标会非常清晰：不是“想办法处理 token”，而是“把 token 序列变成你刚定义好的 AST”。
