Lesson 0030

扫描 fn 关键字

这一课只做 scanner：让语言第一次认识 fn 这个关键字，并把它产出为一个 token。

这一步为什么重要： fn 将是函数字面量的入口关键字，和 true、false、match 一样，先让 scanner 识别它。后续 parser 和 evaluator 才能基于 Token::Fn 构建函数表达式的语法树和求值。

本课范围

  • 给 Token 新增 Fn 变体
  • 在 scanner 的 flush 逻辑中添加 fn 关键字识别
  • 新增 scanner 测试

这一课不做：

  • 不改 parser
  • 不改 AST
  • 不改 evaluator
  • 不支持函数字面量语法
  • 不支持函数调用

这节课真正新增的知识点

scanner 已经处理过很多关键字了：let、print、true、false、match、_。

新增 fn 的模式完全一样：在 flush 的时候检查 value 缓冲区内容，如果是 "fn" 就 push Token::Fn。

但和之前不一样的是：这次不是在做完一个功能后才加关键字，而是在为未来的功能表做铺垫。这是一种常见的语言扩展节奏——先把关键字扫出来，下一课再给它语义。

推荐的最小实现方向

  1. 在 Token 中加入 Fn
  2. 在 scanner 的 flush 闭包里添加 "fn" => Token::Fn
  3. 在 scanner 末尾的收尾逻辑里也添加 "fn" => Token::Fn
  4. 新增测试验证

目标输入 / token 输出

输入:
fn x { x + x }

目标 token:
[
  Fn,
  Ident("x"),
  LBrace,
  Ident("x"),
  Plus,
  Ident("x"),
  RBrace,
]
输入:
let add = fn(x, y) { x + y };

目标 token:
[
  Let,    Ident("add"), Equal,
  Fn,     LParen,       Ident("x"), Comma, Ident("y"), RParen,
  LBrace, Ident("x"),   Plus,       Ident("y"), RBrace, Semicolon,
]

完成标准

  • Token 中已新增 Fn
  • scanner 的 flush 和末尾收尾逻辑都支持 "fn"

  • parser / AST / evaluator 暂时不改
提醒： 这节课的胜利条件不是函数已经能运行，而是"scanner 已经认识 fn 关键字"。
参考来源：
  • Crafting Interpreters — Scanning：scanner 负责把保留字识别为独立 token，不交给 parser 做字符串比较。
  • Rust Reference — Keywords：关键字是 reserved 的标识符，在词法层就已经区分开了。
