Lesson 0013

让 parser 先会处理括号表达式

这一课只教一件事：当表达式被 ( 和 ) 包起来时，parser 不再把它当成噪音，而是真正把里面的表达式解析出来。

和你的 mission 的关系： 你已经有最小 lexer、AST、parser、evaluator 和变量环境，但表达式解析还太脆弱。括号是 Phase 0 表达式系统迈向更真实输入的第一步，它能为后面的优先级处理打基础，又不会像完整优先级那样一次跨太大。

今天只做这一件事

这节课不要尝试一次解决完整优先级系统。范围严格控制在：

  • 让 parser 能识别以 ( 开头的最小分组表达式
  • 让 parser 继续解析括号内部的表达式
  • 让 parser 在合适位置消费掉对应的 )
  • 让这类表达式能接进已有的 let / print

也就是说，本课先打通：

print (1 + 2);

并让它能走完 parse，再交给现有 evaluator 执行。

本课要建立的直觉

括号本身不是一个值，括号的作用是：

  • 告诉 parser：这里有一个“被包起来的表达式”
  • 要求 parser 先把里面那一段作为一个整体解析出来

这和 Crafting Interpreters 对表达式解析的讲法一致：分组不是新值类型，而是改变 parser 理解输入边界的方式。

推荐的最小实现方向

你当前的 `parse_expr()` 已经会处理最小 `atom op atom`。这节课最自然的切入口是：

  1. 当当前 token 是 Token::LParen 时，不要直接报错
  2. 先跳过左括号
  3. 递归调用一次表达式解析，去解析括号里面的内容
  4. 确认后面有一个 Token::RParen
  5. 返回括号内部解析出来的那个表达式

注意：这节课的关键不是发明很多新 AST，而是让 parser 学会“把括号里的表达式当作一个整体”。

本课目标输入 / 输出

输入程序:
print (1 + 2);

目标：
parser 不报错
program 能成功进入 evaluator
最终输出 3
输入程序:
let x = (1 + 2);
print x;

目标：
let 的右侧表达式能被正确解析
最终输出 3

完成标准

  • parser 能处理最小括号表达式，如 (1 + 2)
  • 这个能力已经接进 let 或 print
  • 你已经用真实输入验证过至少一个括号样例

这节课故意不做什么

  • 先不做完整运算符优先级体系
  • 先不做任意复杂嵌套表达式的完美错误恢复
  • 先不引入新的语句种类
  • 先不重构整个 parser 结构

这一课的目标就是：让 parser 对括号不再无能为力。

卡住时优先检查这 4 件事

  • 你是不是在遇到 LParen 时仍然走进了原来的“无效 token”分支？
  • 你是不是解析完括号内部表达式后，忘了再消费一个 RParen？
  • 你是不是把括号 token 当成普通操作符来处理了？
  • 你是不是一边做括号，一边又开始重写完整优先级系统了？
本课后的观察点： 完成这一课后，你的 parser 会第一次真正具备“递归地解析一段被包起来的子表达式”的能力。这会让后续继续做表达式优先级时轻松很多。
